# ── Python ──────────────────────────────────────────────────────────────────
PYTHON │ A default argument is evaluated once at def-time, so `def f(x, seen=[])` shares one list across every call — the canonical accidental-global bug.
PYTHON │ CPython interns ints −5..256 and short identifier-like strings, so `is` "works" on them by coincidence and breaks silently at 257 — never compare values with `is`.
PYTHON │ The GIL is released around blocking I/O and inside many C extensions, so threads help I/O- and C-bound work; pure-Python CPU loops scale only on the free-threaded (3.13t) build.
PYTHON │ `__slots__` drops the per-instance `__dict__`, cutting memory ~40-50% for many small objects and speeding attribute access — at the cost of dynamic attributes and some multiple inheritance.
PYTHON │ `functools.lru_cache` on a method holds a strong ref to `self` through the args key, so the cache pins every instance forever — a classic silent memory leak.
PYTHON │ An un-awaited blocking call (a sync DB driver, `time.sleep`, CPU work) stalls the entire asyncio event loop, not just one task — cooperative scheduling has no preemption.
PYTHON │ A generator keeps its whole stack frame — and every local it references — alive between `next()` calls, so a paused generator can pin a large object until it is exhausted or collected.

# ── Systems & concurrency ───────────────────────────────────────────────────
SYSTEMS │ False sharing: two cores writing different variables that land on the same 64-byte cache line serialize through cache coherency; pad hot per-thread fields to a line to fix it.
SYSTEMS │ A page fault served from disk costs ~10ms vs ~100ns for RAM — a ~100,000× cliff, which is why touching swapped-out memory makes a process feel hung.
SYSTEMS │ `mmap` defers I/O to fault time, so a "pure memory" read can secretly block on disk — invisible latency hiding inside an array access.
SYSTEMS │ A `write()` returns once data is in the page cache, not on disk; only `fsync` makes it durable, so a crash in between silently loses "written" data.
CONCURRENCY │ Without acquire/release barriers a store on one core can be seen out of order by another — which is exactly why naive double-checked locking is broken.
CONCURRENCY │ Branch misprediction costs ~15-20 cycles, so sorting data before a branchy loop can make it several times faster purely by making the branch predictable.
PERFORMANCE │ Amdahl's law: if 10% of the work is serial, infinite cores buy at most a 10× speedup — killing the last serial bottleneck beats adding workers.
PERFORMANCE │ Tail latency compounds under fan-out: a call slow 1% of the time, made 100× per request, is slow ~63% of the time (1 − 0.99¹⁰⁰).

# ── LLMs & ML ───────────────────────────────────────────────────────────────
LLM │ The KV cache trades VRAM for compute — storing past keys/values makes each new token O(seq) instead of O(seq²) — but it grows linearly with context and ends up dominating memory.
LLM │ RoPE encodes position by rotating query/key vectors by an angle proportional to index, so relative position drops out of the dot product — and rescaling its frequencies extends context length.
LLM │ Attention is O(n²) compute; FlashAttention keeps the same math but tiles it in SRAM so it never materializes the n×n matrix in HBM, turning memory from O(n²) to O(n).
LLM │ Speculative decoding lets a small draft model propose k tokens that the big model verifies in one parallel pass — a pure latency win that provably preserves the output distribution.
LLM │ A Mixture-of-Experts routes each token to a few experts, so a 100B-param model may activate only ~10B per token — big capacity at small per-token cost, bounded by routing and memory.
LLM │ Int8 quantization is nearly free for quality because LLM inference is memory-bandwidth-bound, not compute-bound — you simply move fewer bytes per token; int4 trades a little accuracy for more.
LLM │ Tokenization is why models miscount letters: "strawberry" is ~3 tokens, so counting its r's asks the model to introspect sub-token structure it was never shown.
LLM │ Temperature scales logits before softmax (T<1 sharpens, T>1 flattens), but top-p/top-k truncation usually governs coherence more than temperature does.

# ── Databases & Postgres ────────────────────────────────────────────────────
POSTGRES │ MVCC makes an UPDATE write a new tuple and mark the old one dead, so update-heavy tables bloat with dead rows until VACUUM reclaims them.
POSTGRES │ The planner skips an index once it estimates it would read most of the table — past ~5-10% selectivity a sequential scan beats random index lookups.
POSTGRES │ `count(*)` is slow because MVCC visibility is per-row: it must scan to count live tuples rather than read a stored counter.
POSTGRES │ A B-tree on (a, b) serves `a` and `a,b` but not `b` alone — leftmost-prefix means composite-index column order is a design choice, not cosmetic.
POSTGRES │ Each connection is a forked backend process costing megabytes, so Postgres degrades past a few hundred; a pooler multiplexing to ~20 backends usually beats raising the limit.
DATABASES │ REPEATABLE READ still allows write skew — two txns each reading a constraint then writing disjoint rows can both commit and violate it; you need SERIALIZABLE or explicit locks.
DATABASES │ Every index speeds reads but taxes each write and adds deadlock surface, so over-indexing a write-heavy table is a net loss.

# ── Networking ──────────────────────────────────────────────────────────────
NETWORKING │ TCP slow-start ramps throughput over several round-trips, so connection reuse (keep-alive, HTTP/2 multiplexing) dominates latency for many small requests.
NETWORKING │ Nagle's algorithm and delayed ACK can interact to add ~40ms to small writes; `TCP_NODELAY` disables Nagle for latency-sensitive request/response traffic.
NETWORKING │ HTTP/2 multiplexes streams over one TCP connection, so a single lost packet head-of-line-blocks all of them — HTTP/3 over QUIC gives each stream independent loss recovery.
NETWORKING │ TLS 1.3 cuts the handshake to one round-trip (0-RTT on resumption) versus two in 1.2 — a measurable win on every new HTTPS connection.

# ── Frontend / JavaScript / React ───────────────────────────────────────────
JAVASCRIPT │ `==` coerces types with rules surprising enough that `[] == ![]` is true; `===` skips coercion — the reason linters ban `==`.
JAVASCRIPT │ The event loop drains all microtasks (promise callbacks) before the next macrotask (setTimeout), so a runaway promise chain can starve timers and rendering.
REACT │ A `useEffect` with `[]` captures the first render's variables forever, so code inside it reads stale state unless you add deps or route through a ref.
REACT │ Using the array index as a key makes React reuse the wrong DOM node on reorder — corrupting uncontrolled input state — so keys must be stable ids.
REACT │ `useMemo`/`useCallback` cost a comparison plus memory, so memoizing cheap work or non-referential deps is often slower than just recomputing.

# ── Security & correctness ──────────────────────────────────────────────────
SECURITY │ Comparing secrets with `==` leaks length and prefix through timing; use a constant-time compare (`hmac.compare_digest`) for tokens and signatures.
SECURITY │ Bcrypt is intentionally slow and per-hash salted, so equal passwords hash differently and brute force is throttled — never store passwords with a fast hash like SHA-256.
COMPILERS │ A C compiler may assume undefined behavior never happens, so an overflow check written after the overflow gets optimized away — UB can travel backward through your code.
GIT │ A commit's SHA hashes its content plus its parents, so rewriting any old commit changes every descendant hash — which is why rebase rewrites history rather than editing in place.
